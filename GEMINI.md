# 📜 프로젝트 비즈니스 헌법 (GEMINI.md)

본 문서는 **SR(Service Request) 관리 시스템** 프로젝트의 최상위 비즈니스 헌법이다. 에이전트(Antigravity)와 개발자는 본 시스템의 모든 기능을 변경, 추가 또는 디버깅할 때 이 헌법에 정의된 비즈니스 정책을 최우선으로 준수해야 하며, 기술적 코딩 표준은 `.gemini/rules/` 하위의 기술 KI 문서를 따른다.

---

## 1. 사용자 역할 및 권한 규칙 (Role-Based Access Control)

시스템은 철저한 역할 기반 권한 제어(RBAC) 및 고객사 데이터 격리 원칙을 준수한다.

### 1.1. 사용자 역할 정의

- **시스템 관리자 (SYSTEM_ADMIN)**: 전체 시스템 및 모든 고객사의 데이터를 조회, 생성, 수정, 삭제(CRUD)할 수 있으며, 시스템 설정과 전체 감사 로그 조회가 가능하다.
- **고객사 관리자 (CLIENT_ADMIN)**: 자신이 소속된 고객사의 모든 사용자 및 SR을 관리할 수 있으나, 삭제(Delete) 권한은 갖지 못한다.
- **개발자 (DEVELOPER)**: 자신에게 배정/할당된 SR에 한하여 조회, 처리, 완료 및 내부 노트/댓글을 작성할 수 있다. 타 개발자에게 할당된 SR은 마음대로 변경할 수 없다.
- **고객사 사용자 (CLIENT_USER)**: 자신이 속한 고객사의 SR을 신청(생성)할 수 있으며, 본인이 작성한 SR에 대해서만 조회 및 공개 댓글 작성이 가능하다.

### 1.2. 데이터 격리 및 감사 원칙

- **고객사 간 격리 (Multi-Tenant Isolation)**: `SYSTEM_ADMIN`을 제외한 모든 사용자는 자신이 명시적으로 소속되거나 할당받지 않은 타 고객사의 SR 정보, 사용자 정보 및 통계 데이터에 접근할 수 없다. API 및 Server Actions에서 이 데이터 격리가 누출되지 않도록 철저히 검증한다.
- **민감 데이터 보호**: 권한 변경, 삭제 등의 민감 행위는 반드시 Audit Log(감사 로그)로 영구 저장되어야 한다.

## 2. SR 라이프사이클 및 상태 전이 규칙 (SR State Machine)

모든 Service Request는 정해진 라이프사이클 상태 기계를 따르며, 조건 없는 상태 도약은 불가능하다.

```
[신청 (REQUESTED)] ──> [접수 (ACCEPTED)] ──> [진행중 (IN_PROGRESS)] ──> [완료 (COMPLETED)] ──> [확인완료 (CLOSED)]
        │                    │
        └─── 거절 (REJECTED) ┘ (신청/접수 단계에서 반려 가능)
```

- **상태 변경 필수 조건**:
  - `신청` 또는 `접수` 단계에서 `거절 (REJECTED)` 상태로 변경할 때는 **거절 사유(Rejected Reason)**가 반드시 폼 및 DB 데이터로 함께 기입되어야 한다.
  - `진행중`에서 `완료 (COMPLETED)` 상태로 변경할 때는 **완료 내용(Completion Notes)**이 누락 없이 상세하게 작성되어야 한다.
- **보류 (HOLD)**: 진행 중 외부 요인(고객사 응답 대기 등)으로 인해 보류 상태로 전이할 경우, **보류 사유**와 **예상 해제일**을 명시한다.
- **재오픈 (REOPENED)**: `확인완료(CLOSED)` 혹은 `완료(COMPLETED)` 처리 후 7일 이내에만 1회에 한하여 재오픈이 가능하며, **재오픈 사유**를 필수로 기록하고 다시 `진행중` 상태로 전이시킨다.

## 3. SLA 및 우선순위 정책 (SLA & Priority)

SR 신청 시 우선순위는 기본적으로 '중(Medium)'으로 지정되며, 각 우선순위에 따른 접수 및 처리 목표 시간(SLA) 정책은 다음과 같다. 이를 초과할 시 모니터링 시스템에서 경고 알림을 발생시킨다.

| 우선순위            | 접수 목표 시간 | 처리 완료 목표 시간 |
| :------------------ | :------------- | :------------------ |
| **긴급 (Critical)** | 1시간 이내     | 4시간 이내          |
| **높음 (High)**     | 2시간 이내     | 24시간 이내         |
| **중간 (Medium)**   | 24시간 이내    | 48시간 이내         |
| **낮음 (Low)**      | 48시간 이내    | 1주일 이내          |

## 4. 자동 알림 및 연동 정책 (Notification System)

모든 주요 상태 변화는 비동기식 이벤트 알림을 유발해야 한다.

- **알림 채널**: 사용자 설정에 따라 이메일 및 매터모스트(Mattermost) 개인 DM/공용 채널로 발송된다.
- **신청 단계**: SR이 등록되면 해당 고객사 담당자에게 즉시 알림이 발송되어야 한다.
- **배정 단계**: SR 담당자가 지정 또는 변경되면 신규 배정된 담당자에게 알림이 전송되어야 한다.
- **완료 및 거절 단계**: SR 신청자가 결과를 확인할 수 있도록 즉시 완료/거절 알림을 수신자 이메일로 전송한다.
- **알림 폴백**: 매터모스트 알림이 발송 실패할 경우 자동으로 이메일 알림으로 폴백(Fallback)되어야 하며, 발송 로그는 데이터베이스에 기록되어야 한다.
