# SR(Service Request) 관리 시스템 사용자 및 운영 매뉴얼

본 문서는 **SR(Service Request) 관리 시스템**의 통합 사용자 및 운영 매뉴얼입니다. 각 화면별 기능, 주요 UI 구성요소, 역할 기반 권한 제어(RBAC), 그리고 비즈니스 제약조건(SLA, 상태 전이 등)을 명확하게 정의하여 사용자와 운영자의 편의를 돕습니다.

---

## 📚 매뉴얼 활용 방법 및 이미지 등록 안내

본 매뉴얼은 각 화면의 주요 요소와 제약조건을 설명하기 위해 **화면 캡처 플레이스홀더**를 제공합니다. 시스템 관리자 또는 운영자는 실제 작동 중인 화면을 캡처한 후, 다음 규칙에 맞게 이미지를 저장하시면 문서에 자동으로 이미지가 반영됩니다.

1. **이미지 저장 경로**: `d:/project/sr/public/images/manual/`
2. **권장 이미지 포맷**: `PNG` (배경 투명 또는 시스템 테마와 조화로운 사양)
3. **파일명 지정**: 아래 화면별 섹션에 정의된 이미지 파일명(예: `01_login.png`)과 대소문자를 구분하여 동일하게 저장합니다.

## 👥 사용자 역할(Role) 및 데이터 격리 원칙

시스템은 철저한 역할 기반 권한 제어(RBAC) 및 고객사 데이터 격리 원칙을 준수합니다.

| 역할 (Role)                         | 주요 권한 범위                                                                | 데이터 접근 제한 (격리 원칙)                                                                                                     |
| :---------------------------------- | :---------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------- |
| **시스템 관리자<br>(SYSTEM_ADMIN)** | 시스템 설정, 전체 고객사 및 사용자 관리, 전체 감사 로그(Audit Log) 조회       | 모든 고객사 및 사용자의 데이터를 제한 없이 조회 및 수정할 수 있습니다.                                                           |
| **고객사 관리자<br>(CLIENT_ADMIN)** | 소속 고객사의 사용자 및 SR 조회/관리, 배정 변경                               | 본인 고객사 소속 데이터만 조회 가능하며, 타 고객사 데이터에는 접근이 불가능합니다. **데이터 삭제(Delete) 권한은 갖지 못합니다.** |
| **개발자<br>(DEVELOPER)**           | 본인에게 배정된 SR 조회, 처리 완료, 상태 전이, 내부 노트/댓글 작성            | 본인에게 배정된 SR만 관리할 수 있습니다. 타 개발자에게 할당된 SR은 강제로 수정할 수 없습니다.                                    |
| **고객사 사용자<br>(CLIENT_USER)**  | 본인 소속 고객사의 SR 신청, 본인이 작성한 SR 진행 상황 조회 및 공개 댓글 작성 | 자신이 속한 고객사 이외의 타 고객사 정보, 사용자 정보 및 통계 데이터에 접근할 수 없습니다.                                       |

## 🖥️ 화면별 기능 가이드

### 1. 로그인 화면 (Login)

- **화면 경로**: `/login`
- **사용 대상**: 모든 사용자
- **화면 캡처**:
  ![로그인 화면 캡처](file:///d:/project/sr/public/images/manual/01_login.png)

#### 주요 UI 구성요소 및 설명

- **이메일 주소**: 사용자의 고유 가입 이메일을 입력합니다. (로그인 ID로 사용)
- **비밀번호**: 가입 시 설정한 비밀번호를 입력합니다.
- **로그인 상태 유지**: 체크 시 브라우저 쿠키를 통해 세션을 유지합니다.
- **회원가입 링크**: 아직 계정이 없는 사용자를 위한 회원가입 페이지(`/register`)로 이동합니다.

#### 제약 조건 및 비즈니스 예외 처리

- **인증 정보 보호**: 로그인 정보 송신 시 HTTPS 보안 프로토콜을 사용하며, 비밀번호는 단방향 해시로 보호됩니다.
- **휴면 계정 제한**: 6개월 이상 시스템 미사용으로 인해 '휴면' 상태가 된 계정은 로그인 시도 시 관리자 재활성화 승인 안내 팝업이 노출됩니다.

### 2. 회원가입 화면 (Register)

- **화면 경로**: `/register`
- **사용 대상**: 신규 사용자
- **화면 캡처**:
  ![회원가입 화면 캡처](file:///d:/project/sr/public/images/manual/02_register.png)

#### 주요 UI 구성요소 및 설명

- **사용자명**: 실제 사용자 이름을 입력합니다.
- **이메일 주소**: 고유한 이메일 주소를 입력하며, 가입 후 이메일 인증이 진행됩니다.
- **비밀번호/비밀번호 확인**: 영문, 숫자, 특수문자를 혼합한 보안 비밀번호를 입력합니다.
- **고객사 검색 및 선택**: 소속될 고객사 명을 입력하고 자동완성 제안 목록에서 선택합니다.
- **부서/팀 선택**: 선택한 고객사 산하에 등록되어 있는 부서/팀을 선택합니다.
- **휴대폰 번호**: 본인 인증 및 비상 연락 수단으로 활용될 전화번호를 입력합니다.

#### 제약 조건 및 비즈니스 예외 처리

- **이메일 고유성**: 이미 가입된 이메일은 회원가입 프로세스 진행이 제한됩니다.
- **이메일 인증 링크**: 가입 신청 완료 시 이메일 인증 메일이 발송됩니다. 해당 링크의 **유효 기간은 24시간**이며, 기간 만료 시 재발송을 요청해야 합니다.

### 3. 실시간 통합 대시보드 (Dashboard)

- **화면 경로**: `/dashboard`
- **사용 대상**: SYSTEM_ADMIN, CLIENT_ADMIN, DEVELOPER (권한별 조회 범위 상이)
- **화면 캡처**:
  ![대시보드 화면 캡처](file:///d:/project/sr/public/images/manual/03_dashboard.png)

#### 주요 UI 구성요소 및 설명

- **SR 실시간 상태 현황 카드**: '신청', '접수', '진행중', '보류', '완료', '거절' 상태별 SR의 개수를 표시합니다.
- **SLA 경고 보드**: 접수 및 완료 목표 시간(SLA)이 초과되었거나 초과가 임약한 '긴급(Critical)' / '높음(High)' SR 목록을 강조 표시합니다.
- **최근 등록된 SR**: 최근 24시간 이내에 접수된 신규 요청 리스트를 보여줍니다.
- **담당자별/고객사별 통계 차트**: 담당자 업무 과중도와 고객사별 청구 건수를 가시화한 차트 컴포넌트입니다.

#### 제약 조건 및 비즈니스 예외 처리

- **데이터 격리 적용**:
  - `CLIENT_ADMIN`은 본인 소속 고객사의 통계 데이터만 조회됩니다.
  - `DEVELOPER`는 대시보드 내에서 자신에게 배정된 SR의 통계 및 전체 현황 중 본인 관여 건 위주로 필터링되어 보입니다.

- **화면 경로**: `/srs`
- **사용 대상**: SYSTEM_ADMIN, CLIENT_ADMIN, DEVELOPER
- **화면 캡처**:
  ![SR 목록 화면 캡처](file:///d:/project/sr/public/images/manual/04_sr_list.png)

#### 주요 UI 구성요소 및 설명

- **통합 필터 바**: 고객사명, 진행 상태, 우선순위(긴급, 높음, 중간, 낮음), 등록일자, 담당자명 조건으로 목록을 필터링합니다.
- **검색 창**: 제목 및 내용을 키워드로 검색합니다.
- **SR 데이터 테이블**: SR 번호, 제목, 신청자, 담당자, 등록일, 우선순위, 현재 상태 정보를 테이블 레이아웃으로 출력합니다.
- **엑셀 내보내기 버튼**: 현재 필터링된 SR 목록을 Excel 파일로 즉시 다운로드합니다.

- **데이터 접근 제한**: `CLIENT_USER` 권한의 사용자는 이 메뉴에 직접 접근할 수 없으며, 본인의 전용 메뉴인 `/my-requests`를 통해서만 자신의 건만 확인할 수 있습니다.

### 5. SR 상세 화면 및 상태 관리 (SR Detail & State Machine)

- **화면 경로**: `/srs/[id]`
- **사용 대상**: SYSTEM_ADMIN, CLIENT_ADMIN, DEVELOPER, CLIENT_USER(본인 소유 건에 한함)
- **화면 캡처**:
  ![SR 상세 화면 캡처](file:///d:/project/sr/public/images/manual/05_sr_detail.png)

#### 주요 UI 구성요소 및 설명

- **SR 정보 요약**: 제목, 설명, 요청 분류, 우선순위, 첨부파일 목록을 확인할 수 있습니다.
- **상태 변경 컨트롤러**: 권한에 따라 현재 SR의 단계를 전이(Transition)시킵니다.
- **우선순위 설정**: 긴급, 높음, 중간, 낮음의 단계를 선택하여 설정합니다.
- **담당자 지정/변경**: 현재 접수된 요청을 처리할 담당 개발자를 지정합니다.
- **거절 사유/완료 내용 입력창**: 상태 전이에 필요한 텍스트를 기입합니다.
- **댓글 영역**: 공개 댓글(신청자 소통용) 및 내부 노트(개발자 간 내부 커뮤니케이션용)를 작성할 수 있는 영역입니다.

#### 제약 조건 및 비즈니스 예외 처리

- **상태 전이 헌법 필수 준수**:
  - `신청(REQUESTED)` 또는 `접수(ACCEPTED)` 단계에서 `거절(REJECTED)` 상태로 변경할 때는 **거절 사유(Rejected Reason)**가 필수로 기입되어야만 저장이 가능합니다.
  - `진행중(IN_PROGRESS)`에서 `완료(COMPLETED)` 상태로 변경할 때는 **완료 내용(Completion Notes)**이 누락 없이 상세하게 작성되어야 저장이 성공합니다.
  - 외부 피드백 대기 등으로 `보류(HOLD)` 상태로 전이할 시에는 **보류 사유**와 **예상 해제일**을 함께 명시해야 합니다.
- **재오픈(REOPENED) 제한**:
  - `완료(COMPLETED)` 또는 `확인완료(CLOSED)` 처리된 건에 대해서는 **최종 처리 후 7일 이내에만 1회에 한하여 재오픈**이 가능합니다.
  - 재오픈 시에는 **재오픈 사유**를 입력해야 하며, 상태는 다시 `진행중(IN_PROGRESS)`으로 강제 전이됩니다.
- **댓글 작성 권한**:
  - `CLIENT_USER`는 일반 공개 댓글만 작성할 수 있으며, 개발자들만 볼 수 있는 **내부 노트(Private Note)**에는 접근하거나 이를 읽을 수 없습니다.

- **사용 대상**: CLIENT_USER (고객사 사용자)
- **화면 캡처**:
  ![나의 요청 목록 화면 캡처](file:///d:/project/sr/public/images/manual/06_my_requests.png)

### 7. 고객사 관리 화면 (Client Management)

- **화면 경로**: `/clients`
- **사용 대상**: SYSTEM_ADMIN (시스템 관리자)
- **화면 캡처**:
  ![고객사 관리 화면 캡처](file:///d:/project/sr/public/images/manual/07_clients.png)

#### 주요 UI 구성요소 및 설명

- **고객사 등록/수정 폼**: 고객사명, 고객사 코드(자동생성), 계약 시작일 및 종료일, 주요 연락처를 입력합니다.
- **카테고리 설정 탭**: 해당 고객사 전용 서비스 유형(Technical Support, Billing 등)을 설정합니다.
- **상태 관리**: 활성, 휴면, 종료, 보류 상태를 관리합니다.

#### 제약 조건 및 비즈니스 예외 처리

- **고유성 제약**: 고객사명과 고객사 코드는 DB 전체에서 유일해야 합니다.
- **삭제 불가 원칙**: `CLIENT_ADMIN`은 고객사 설정 수정을 시도할 수 없으며, 모든 고객사 데이터의 영구 삭제는 감사 기록 보호를 위해 철저히 차단되고 `SYSTEM_ADMIN`에 의한 논리 삭제(비활성 처리)만 지원됩니다.

- **화면 경로**: `/users`
- **사용 대상**: SYSTEM_ADMIN, CLIENT_ADMIN (본인 고객사 내)
- **화면 캡처**:
  ![사용자 관리 화면 캡처](file:///d:/project/sr/public/images/manual/08_users.png)

#### 주요 UI 구성요소 및 설명

- **사용자 목록**: 전체 또는 소속 고객사 사용자들의 가입 상태, 역할, 부서, 이메일을 확인합니다.
- **역할 및 권한 지정**: 사용자의 역할(System Admin, Client Admin, Developer, Client User)을 변경 부여합니다.
- **사용자 계정 상태 제어**: 계정을 '비활성화' 하거나 '휴면 해제' 처리합니다.

## ⏱️ SLA 및 우선순위 자동 알림 정책 요약

매뉴얼을 효율적으로 운영하기 위해, 아래의 시스템 자동 SLA 타이머 정책과 알림 전송 타겟을 수시로 검토해 주시기 바랍니다.

### 1. 우선순위별 목표 시간 (SLA)

| 우선순위 (Priority) | 접수 목표 시간 (SLA) | 처리 완료 목표 시간 (SLA) |
| :------------------ | :------------------- | :------------------------ |
| **긴급 (Critical)** | 1시간 이내           | 4시간 이내                |
| **높음 (High)**     | 2시간 이내           | 24시간 이내               |
| **중간 (Medium)**   | 24시간 이내          | 48시간 이내               |
| **낮음 (Low)**      | 48시간 이내          | 1주일 이내                |

- 목표 접수 및 완료 시간이 초과될 경우, 대시보드에 적색 경고가 점멸하며 소속 고객사 담당 개발자에게 즉각 경고 알림이 전송됩니다.

### 2. 자동 알림(Notification) 발생 조건

- **SR 등록 시**: 해당 고객사 담당 관리자(CLIENT_ADMIN 및 배정 개발자)에게 메일 및 웹 푸시가 자동 전송됩니다.
- **담당자 배정 시**: 지정 또는 교체된 개발자에게 "신규 배정 알림"이 즉시 발송됩니다.
- **완료 및 거절 처리 시**: SR을 최초 제기했던 신청자(`CLIENT_USER`)에게 완료 사유 또는 거절 사유가 담긴 요약 알림이 메일로 자동 발송됩니다.
